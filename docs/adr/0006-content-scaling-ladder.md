# ADR-0006: 콘텐츠 스케일링 사다리 — git이 DB다

**상태:** 승인됨

**날짜:** 2026-07-20

## 맥락

공개 기록이 늘면 사이트가 무거워지지 않는지, 별도 데이터베이스를 미리 구축해야 하는지 질문이 나왔다. 2026-07-20 실측 기준: 공개 노트 10편 전문을 담은 `content/public/index.json`은 16KB(gzip 6KB), `graph.json`은 5.5KB(gzip 1.2KB), 두 파일을 포함한 사이트 JS 번들 전체가 gzip 20KB다. 노트 1편당 전송 비용은 약 0.6KB로, 별자리 이미지 한 장(수백 KB)보다 훨씬 작다. 늘어나는 것은 코드가 아니라 번들에 실리는 데이터이며, 현재 규모에서는 무시할 수 있는 수준이다.

한편 이 시스템의 발행 게이트는 "발행 = 검증된 커밋 + 배포"라는 사실 위에 서 있다. 승인 증적, 민감 패턴 검사, 결정론적 artifact(`index.json`, `graph.json`)는 모두 git 히스토리로 감사할 수 있다. 별도 DB를 권위 원천으로 도입하면 게이트를 우회해 콘텐츠를 넣을 수 있는 뒷문이 생기고, "한 정보 유형에는 한 개의 권위 원천"이라는 차터 원칙이 깨진다.

## 결정

- **공개 콘텐츠의 권위 원천은 계속 Markdown frontmatter + git이다.** 어떤 규모에서도 DB에 직접 콘텐츠를 작성하지 않는다.
- 별도 DB는 미리 구축하지 않고, 아래 사다리의 트리거가 실측으로 확인될 때만 다음 계단을 밟는다.

| 계단 | 트리거 (실측 기준) | 조치 | DB |
| --- | --- | --- | --- |
| 0. 현재 | — | 아무것도 하지 않는다 | 없음 |
| 1. 번들 분리 | 노트 약 50편 또는 `index.json` gzip 50KB 초과 | JSON을 JS 번들 import에서 분리해 런타임 fetch 정적 파일로 전환. 목록·검색용 경량 index와 본문을 분리하고 본문은 모달을 열 때 로드 | 없음 (정적 파일 유지) |
| 2. 검색 인덱스 | 노트 약 150편 또는 검색 입력 지연 체감 | 키 입력마다 전체 본문을 토큰화하는 현재 방식을, 빌드 타임에 사전 계산한 검색 인덱스로 교체 | 없음 |
| 3. 파생 read model | 노트 약 500편 이상, 또는 배포와 분리된 발행·조회수·댓글 같은 동적 기능이 실제로 필요할 때 | Cloudflare D1(또는 KV)을 도입하되 **CI가 git에서 생성·동기화하는 읽기 전용 사본으로만** 쓴다. 스키마와 동기화 스크립트는 이 저장소에서 관리한다 | 파생 전용 |

- 각 계단은 이전 계단의 트리거가 실측으로 확인된 뒤에만 진행하며, 추측으로 앞당기지 않는다.
- 계단 3에 도달하더라도 발행 절차(초안 → 검토 증적 → 승인 → 커밋 → 배포)는 변하지 않는다. DB 장애 시 사이트는 정적 artifact로 폴백할 수 있어야 한다.

## 결과

지금 리듬(주 1~2편)으로 계단 1까지 약 반년의 여유가 있다. 그동안 콘텐츠 축적에 집중하고 인프라 결정을 반복하지 않는다. 이 문서의 수치를 벗어나는 실측이 나오면 그때 해당 계단만 실행하고, 결정이 바뀌면 새 ADR로 대체한다.
